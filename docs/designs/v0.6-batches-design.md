# v0.6.0 Batches API — Design

**Bead**: `claude-messaging-pharo-53h`
**Status**: PROPOSED
**Author**: Claude Agento
**Date**: 2026-04-23
**Target release**: v0.6.0

---

## 1. Overview

The Message Batches API is Anthropic's async submission surface for the
Messages API. Rather than calling `POST /v1/messages` once per prompt and
waiting synchronously for each response, you submit up to tens of thousands
of requests in a single `POST /v1/messages/batches` and poll a batch object
until `processingStatus` reaches `ended`. Results arrive as a
newline-delimited JSON (JSONL) file streamed from a URL stored in the batch
object.

Key differences from the synchronous Messages API:

| Dimension | Messages | Batches |
|-----------|----------|---------|
| Endpoint | `POST /v1/messages` | `POST /v1/messages/batches` |
| Response timing | Synchronous (blocks) | Async — poll until `ended` |
| Result format | One JSON body | JSONL stream, one line per request |
| Result ordering | N/A | Not guaranteed; match via `custom_id` |
| Expiry | None | 24 hours after creation |
| Beta header | Optional | `message-batches-2024-09-24` required |
| Delete | N/A | After processing ends; cancel first if in-progress |

The Batches API shares the Messages request shape exactly —
`BetaMessageBatchNewParamsRequestParams` in the Go SDK wraps the same fields
as `MessageNewParams`. In Pharo terms, a batch submission is a collection of
`ClaudeMessageRequest` objects each tagged with a `customId` string.

Every operation targets the `/v1/messages/batches` endpoint family with a
mandatory beta header (`message-batches-2024-09-24`). The Pharo SDK injects
this header automatically via a dedicated HTTP-client factory on
`ClaudeClient`, matching the pattern already used by `newFileHttpClient` for
the Files API.

## 2. Package layout

**Decision**: new package `Claude-Messaging-Batches`.

Per ADR-40, Batches is Messages-API-adjacent — it wraps the same
`ClaudeMessageRequest` shape — so it belongs to the `Claude-Messaging-*`
family, not `Claude-ManagedAgents-*`. The resource is distinct enough from
the synchronous Messages surface that it earns its own package rather than
being absorbed into `Claude-Messaging-Client`. This mirrors how `Files` and
`Skills` each got their own packages despite being lightweight.

Per ADR-42, `ClaudeClient` is extended with batch methods via Pharo extension
methods defined in `Claude-Messaging-Batches`. The extension method category
is `*Claude-Messaging-Batches`.

**New packages:**

| Package | Type | Contents |
|---------|------|----------|
| `Claude-Messaging-Batches` | Production | All batch types + `ClaudeClient` extensions |
| `Claude-Messaging-Batches-Tests` | Test | Unit tests for every class, mock-based integration |

**Baseline dependency order** (appended to `BaselineOfClaudeMessaging`):

```text
Claude-Messaging-Batches
  requires: Claude-Messaging-Types   (ClaudeMessageRequest reused as per-item params)
            Claude-Messaging-Client   (ClaudeClient host class for extensions)
            Claude-Messaging-Errors   (ClaudeApiError hierarchy)
```

`Claude-Messaging-Batches-Tests` requires `Claude-Messaging-Batches` and
`Claude-Messaging-Client-Tests` (for `ClaudeMockServer`).

## 3. Type list

All types live in `Claude-Messaging-Batches`.

### 3.1 Response types

**`ClaudeBatch`** — the primary batch object returned by create, get, cancel.
Corresponds to `BetaMessageBatch` in the Go SDK.

Slots:

- `id` — String; opaque batch identifier
- `processingStatus` — String; one of `'in_progress'`, `'canceling'`, `'ended'`
- `requestCounts` — `ClaudeBatchRequestCounts`
- `createdAt` — String; RFC 3339
- `expiresAt` — String; RFC 3339; always 24 h after `createdAt`
- `endedAt` — String or nil; RFC 3339; set when `processingStatus = 'ended'`
- `cancelInitiatedAt` — String or nil; RFC 3339
- `archivedAt` — String or nil; RFC 3339
- `resultsUrl` — String or nil; URL to JSONL results file; set once `ended`
- `type` — literal `'message_batch'`

**`ClaudeBatchRequestCounts`** — tallies per-result-type counts. Corresponds
to `BetaMessageBatchRequestCounts`.

Slots: `processing`, `succeeded`, `errored`, `canceled`, `expired` — all
Integer. All zero until the batch reaches `ended`.

**`ClaudeDeletedBatch`** — confirmation object returned by delete.
Corresponds to `BetaDeletedMessageBatch`.

Slots:

- `id` — String
- `type` — literal `'message_batch_deleted'`

**`ClaudeBatchPage`** — paginated list for `listBatches`. Cursor-ID
pagination (same shape as `ClaudeFilePage`).

Slots: `data` (Array of `ClaudeBatch`), `hasMore` (Boolean), `firstId`
(String or nil), `lastId` (String or nil).

**`ClaudeBatchResult`** — one entry in the JSONL results stream. Corresponds
to `BetaMessageBatchIndividualResponse`.

Slots:

- `customId` — String; matches the `customId` on the submitted request
- `result` — one of the four result-outcome objects below

**`ClaudeBatchSucceededResult`** — wraps a `ClaudeMessage`. Corresponds to
`BetaMessageBatchSucceededResult`.

Slots: `type` (literal `'succeeded'`), `message` (`ClaudeMessage`).

**`ClaudeBatchErroredResult`** — wraps an error payload. Corresponds to
`BetaMessageBatchErroredResult`.

Slots: `type` (literal `'errored'`), `error` (`ClaudeBatchErrorPayload`).

**`ClaudeBatchCanceledResult`** — terminal state; no payload beyond the type.
Corresponds to `BetaMessageBatchCanceledResult`.

Slots: `type` (literal `'canceled'`).

**`ClaudeBatchExpiredResult`** — terminal state; no payload. Corresponds to
`BetaMessageBatchExpiredResult`.

Slots: `type` (literal `'expired'`).

**`ClaudeBatchErrorPayload`** — the `error` field inside `ClaudeBatchErroredResult`.
Mirrors `BetaErrorResponse`.

Slots: `type` (String), `message` (String).

### 3.2 Result union dispatch

The Go SDK uses a union type for the `result` field. In Pharo, `ClaudeBatchResult`
holds the raw result dictionary and deserializes it via a class-side
`fromJson:result:` factory that reads the `type` key and dispatches to the
correct outcome class. No conditional chains in the domain logic; callers use
polymorphism:

```smalltalk
"Sketch — not implementation"
ClaudeBatchResult class >> resultFromJson: aDict
    | type |
    type := aDict at: 'type'.
    type = 'succeeded' ifTrue: [ ^ ClaudeBatchSucceededResult fromJson: aDict ].
    type = 'errored'   ifTrue: [ ^ ClaudeBatchErroredResult fromJson: aDict ].
    type = 'canceled'  ifTrue: [ ^ ClaudeBatchCanceledResult fromJson: aDict ].
    type = 'expired'   ifTrue: [ ^ ClaudeBatchExpiredResult fromJson: aDict ].
    ^ ClaudeBatchErroredResult unknown: type
```

Each outcome class answers `isSucceeded`, `isErrored`, `isCanceled`,
`isExpired` — one returns true, the others false. Callers dispatch without
conditionals.

### 3.3 Request types

**`ClaudeBatchRequest`** — one item in a batch submission. Corresponds to
`BetaMessageBatchNewParamsRequest`.

Slots:

- `customId` — String; developer-assigned; must be unique within the batch
- `params` — `ClaudeMessageRequest`; the full message request for this item

**`ClaudeBatchCreateParams`** — the body for `POST /v1/messages/batches`.
Corresponds to `BetaMessageBatchNewParams`.

Slots: `requests` — OrderedCollection of `ClaudeBatchRequest`.

Builder methods:

```smalltalk
"Sketch"
ClaudeBatchCreateParams new
    addRequest: (ClaudeBatchRequest customId: 'req-1' params: aMessageRequest);
    yourself
```

**`ClaudeBatchListParams`** — query parameters for `listBatches:`. Corresponds
to `BetaMessageBatchListParams`.

Slots: `limit` (Integer, default nil → API default of 20), `afterId` (String
or nil), `beforeId` (String or nil).

## 4. `ClaudeClient` extension methods

All five methods are defined as extension methods in `Claude-Messaging-Batches`
under category `*Claude-Messaging-Batches`. They use an internal private
method `newBatchHttpClient` that injects the required beta header
(`message-batches-2024-09-24`) before every request, matching the pattern of
`newFileHttpClient` in the Files package.

**`createBatch: aClaudeBatchCreateParams`** — POST `/v1/messages/batches`.
Returns `ClaudeBatch`. Raises `ClaudeInvalidRequestError` if `requests` is
empty.

**`getBatch: batchId`** — GET `/v1/messages/batches/{id}`. Returns
`ClaudeBatch`. Idempotent; safe to call repeatedly for polling.

**`listBatches`** — GET `/v1/messages/batches` with default params. Returns
`ClaudeBatchPage`.

**`listBatches: aClaudeBatchListParams`** — parameterized variant. Returns
`ClaudeBatchPage`. Clients iterate by feeding `page lastId` into the next
call's `afterId`.

**`cancelBatch: batchId`** — POST `/v1/messages/batches/{id}/cancel`. Returns
`ClaudeBatch` with `processingStatus = 'canceling'`. Can only succeed while
`processingStatus = 'in_progress'`.

**`deleteBatch: batchId`** — DELETE `/v1/messages/batches/{id}`. Returns
`ClaudeDeletedBatch`. Only valid once `processingStatus = 'ended'` (the API
returns 400 otherwise).

**`streamBatchResults: batchId do: aBlock`** — GET
`/v1/messages/batches/{id}/results` with `Accept: application/x-jsonl`. Reads
the JSONL response line-by-line; parses each line as a `ClaudeBatchResult`;
calls `aBlock` with each result. Returns an OrderedCollection of all
`ClaudeBatchResult` objects once the stream is exhausted. Raises
`ClaudeBatchNotEndedError` (see §7) if `resultsUrl` is not yet set.

Convenience polling method (on `ClaudeClient`, same extension package):

**`pollBatch: batchId untilEndedEvery: seconds`** — loops on `getBatch:`,
sleeping `seconds` between polls, until `processingStatus = 'ended'`. Returns
the final `ClaudeBatch`. Does not fetch results — callers follow up with
`streamBatchResults:do:`.

## 5. JSON serialization

All types follow ADR-5: no reflection. Explicit `asJson` / `fromJson:` with
`jsonKeyMap` for snake_case ↔ camelCase mapping.

**Wire key map** (selected fields needing translation):

| Smalltalk slot | Wire key |
|---------------|----------|
| `processingStatus` | `processing_status` |
| `requestCounts` | `request_counts` |
| `createdAt` | `created_at` |
| `expiresAt` | `expires_at` |
| `endedAt` | `ended_at` |
| `cancelInitiatedAt` | `cancel_initiated_at` |
| `archivedAt` | `archived_at` |
| `resultsUrl` | `results_url` |
| `customId` | `custom_id` |
| `hasMore` | `has_more` |
| `firstId` | `first_id` |
| `lastId` | `last_id` |

**Request serialization** — `ClaudeBatchCreateParams >> asJson` produces:

```json
{
  "requests": [
    { "custom_id": "req-1", "params": { <ClaudeMessageRequest wire shape> } }
  ]
}
```

`ClaudeMessageRequest` already implements `asJson` for the Messages API wire
shape. `ClaudeBatchRequest >> asJson` delegates directly.

**Response deserialization** — `ClaudeBatch >> fromJson:` uses `ifAbsent:
[nil]` for optional datetime fields (`endedAt`, `cancelInitiatedAt`,
`archivedAt`, `resultsUrl`) so partial objects during `in_progress` remain
valid.

**JSONL deserialization** — each line in the results stream is a standalone
JSON object. `ClaudeClient >> streamBatchResults:do:` reads lines and calls
`ClaudeBatchResult fromJson: (STONJSON fromString: line)` per line.
`STONJSON` is used throughout the SDK; no new dependency.

## 6. Streaming results (JSONL)

The results endpoint returns `application/x-jsonl`, not SSE. Each response
line is one complete JSON object — a `BetaMessageBatchIndividualResponse` in
the Go SDK vocabulary. The `ClaudeStreamingSocket` and `ClaudeSSEDecoder`
machinery is SSE-specific and is not reused here.

**Approach**: buffered line reader on the raw `ZnClient` response body.

`ClaudeClient >> streamBatchResults:do:` implementation sketch:

```smalltalk
"Sketch — not implementation"
streamBatchResults: batchId do: aBlock
    | batch client response results |
    batch := self getBatch: batchId.
    batch resultsUrl isNil ifTrue: [ ClaudeBatchNotEndedError signal ].
    client := self newBatchHttpClient.
    response := client
        url: batch resultsUrl;
        get;
        response.
    results := OrderedCollection new.
    response entity readStream lines do: [ :line |
        line isEmpty ifFalse: [
            | entry |
            entry := ClaudeBatchResult fromJson: (STONJSON fromString: line).
            results add: entry.
            aBlock value: entry ] ].
    ^ results
```

This avoids a new streaming class. The JSONL response is small enough to fit
in ZnClient's buffer for realistic batch sizes. A future release can add
incremental chunked reading if needed (tracked separately).

**Note on `resultsUrl`**: The Anthropic API only populates `results_url` once
`processing_status = 'ended'`. Callers should confirm ended status (or use
`pollBatch:untilEndedEvery:`) before calling `streamBatchResults:do:`.

## 7. Error handling

### New error class

**`ClaudeBatchNotEndedError`** — raised by `streamBatchResults:do:` when
`resultsUrl` is nil (batch not yet ended). This is a client-side guard, not
an API HTTP error.

```smalltalk
"In Claude-Messaging-Batches"
ClaudeApiError subclass: #ClaudeBatchNotEndedError
    "Raised when streamBatchResults:do: is called on a batch that
     has not yet reached processingStatus = 'ended'."
```

No additional HTTP-status subclasses are needed for Batches. The existing
`ClaudeApiError` hierarchy already covers the HTTP errors the Batches
endpoints can return:

| Scenario | Existing class |
|----------|---------------|
| Batch not found | `ClaudeNotFoundError` (404) |
| Delete while in-progress (not canceled) | `ClaudeInvalidRequestError` (400) |
| Auth failure | `ClaudeAuthError` (401) |
| Rate limit hit on batch submission | `ClaudeRateLimitError` (429) |
| Server overloaded | `ClaudeOverloadedError` (529) |

`ClaudeBatchNotEndedError` inherits from `ClaudeApiError` rather than `Error`
so callers who catch `ClaudeApiError` catch it automatically.

## 8. Test plan

### Unit tests (`ClaudeBatchTest`, `ClaudeBatchResultTest`, etc.)

All in `Claude-Messaging-Batches-Tests`.

- `ClaudeBatchTest` — `fromJson:` round-trips for all field combinations:
  full object (all optional fields present), minimal object (`processingStatus
  = 'in_progress'`, `resultsUrl` nil), status transitions. Verify `isEnded`,
  `isInProgress`, `isCanceling` predicate methods.
- `ClaudeBatchRequestCountsTest` — `fromJson:` with all five count fields; sum
  invariant (sum equals total submitted, though not enforced in SDK).
- `ClaudeDeletedBatchTest` — `fromJson:` and `type` literal.
- `ClaudeBatchPageTest` — `fromJson:` with empty `data`, with multiple entries;
  cursor-ID pagination via `hasMore`, `lastId`.
- `ClaudeBatchResultTest` — `fromJson:` for each of the four outcome variants
  (`succeeded`, `errored`, `canceled`, `expired`). Verify predicate dispatch
  (`isSucceeded` etc.). Verify `ClaudeMessage` deserialization inside
  `ClaudeBatchSucceededResult`.
- `ClaudeBatchCreateParamsTest` — `asJson` with one request; `asJson` with
  multiple requests. Verify `custom_id` and nested `params` keys.
- `ClaudeBatchListParamsTest` — `asQueryString` variants: no params, limit
  only, cursor ID pagination.
- `ClaudeBatchClientTest` — uses `ClaudeMockServer`. Tests:
  - `createBatch:` sends correct beta header and body; returns `ClaudeBatch`.
  - `getBatch:` fetches by ID; returns `ClaudeBatch`.
  - `listBatches` returns `ClaudeBatchPage`.
  - `cancelBatch:` sends correct path; returns `ClaudeBatch`.
  - `deleteBatch:` sends DELETE; returns `ClaudeDeletedBatch`.
  - `streamBatchResults:do:` parses JSONL body line by line; calls block once
    per entry; handles an empty result stream.
  - `streamBatchResults:do:` raises `ClaudeBatchNotEndedError` when
    `resultsUrl` is nil.

### Integration tests (tagged `integration` — excluded from `make test`)

`ClaudeBatchIntegrationTest` — requires live API key:

1. Submit a small batch (2–3 requests with `model: ClaudeModel haiku35`).
2. `pollBatch:untilEndedEvery: 5` until `ended` (≤ 24 h; in practice
   seconds for tiny batches).
3. `streamBatchResults:do:` — verify each `customId` matches a submitted
   request; verify all results have `isSucceeded = true`; verify each
   `ClaudeBatchSucceededResult message` has non-empty `textContent`.
4. `deleteBatch:` — verify `ClaudeDeletedBatch` returned.

Tag: add `#integration` to the test class pragmas so `ClaudeMessagingTestSuite`

## 10. ADR-43 (to be added to DESIGN.md post-implementation)

**ADR-43 — JSONL results via buffered line reader, not `ClaudeStreamingSocket`**

**Status**: PROPOSED

**Decision**: `streamBatchResults:do:` reads the JSONL results body using a
buffered line reader on the ZnClient response entity rather than reusing
`ClaudeStreamingSocket`.

**Rationale**: SSE streaming (used by the synchronous Messages API) requires
chunked transfer-encoding support and incremental event parsing. JSONL
results for Batches are a plain HTTP GET response; the server writes the full
file and closes the connection. ZnClient buffers the body correctly for this
use case. A raw TLS socket would add complexity with no benefit.

**Rejected alternative**: New `ClaudeJSONLDecoder` class mirroring
`ClaudeSSEDecoder`. The batch results endpoint does not use chunked
transfer-encoding, so the complexity of a streaming decoder is unnecessary.
If a future Anthropic endpoint returns incremental JSONL over a long-lived
connection, a decoder class becomes warranted at that point.

**Consequences**: `streamBatchResults:do:` loads the full response into
memory before iteration. For batches with tens of thousands of results, this
can be 10–100 MB. Acceptable for v0.6; a chunked variant can be added if
callers report memory pressure.

      `cancelBatch:`, `deleteBatch:`, `streamBatchResults:do:`,

    - One test class per production class (steps 2–11).

14. **`BaselineOfClaudeMessaging`** update — add
    `Claude-Messaging-Batches` and `Claude-Messaging-Batches-Tests`

**Step ordering rationale**: outcome classes (step 2) have no dependencies;
they can be defined and tested before the container types. `ClaudeBatch`
(step 4) depends on `ClaudeBatchRequestCounts` (step 3). `ClaudeBatchResult`
(step 6) depends on all outcome classes. Extensions (step 12) depend on all
types. Baseline update (step 14) is last — it must name classes that already
exist in Tonel.
